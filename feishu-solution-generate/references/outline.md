## 一、项目概述

### 1.1 项目背景
- 客户企业简介（行业、规模、业务范围）
- 当前面临的业务痛点（用客户能感知的场景描述）
- 已有尝试及效果不理想的原因
- **飞书特有切入点**：客户是否已在使用飞书？用了哪些产品？协作痛点在哪里？

### 1.2 建设目标
- 本次要解决的核心问题（3-5条）
- 飞书生态的天然优势怎么对应这些问题（协作+AI原生一体）

### 1.3 项目范围
- "做什么 / 不做什么（本期）"对比表
- 边界比功能列表更重要

## 二、需求理解与分析

### 2.1 业务流程梳理
- AS-IS → TO-BE 流程对比
- 标注飞书产品在哪些节点介入（如：这个节点用多维表格，那个节点用审批）

### 2.2 角色与场景
- 涉及角色及其飞书使用场景和痛点

### 2.3 关键需求映射
- 客户原话 → 理解后需求 → 应对思路（三列表）
- 有模糊需求标注"待详细调研确认"

## 三、解决方案概要

### 3.1 整体架构蓝图
- 一张图说明系统在飞书生态中的位置
- 结构：业务场景层 → 飞书Aily（AI层：智能伙伴/妙搭OpenClaw/自定义智能体） → 飞书产品矩阵 → 外部接入
- 强调：飞书的AI不是独立平台，是嵌入在文档/消息/会议/表格里的

### 3.2 核心功能模块
每模块按以下结构：

**模块X：[名称]**
- 干什么 + 为什么这么做 + 怎么做
- 关键设计决策（如有特殊选择，解释why）
- 具体用的飞书产品组合（如：多维表格做数据库 + 应用引擎做表单 + 智能伙伴做分类）

### 3.3 关键场景走查
- 选 2-3 个核心场景，用人物+步骤演示
- 例："运营小王在飞书消息里收到一条商机卡片 → 点击查看多维表格详情 → …"

### 3.4 方案价值总览
- Before/After 对比表（业务指标）

## 四、对接与集成方案

### 4.1 飞书能力利用

### 4.2 外部系统对接
- 系统名称 | 对接方式 | 数据方向 | 优先级
- 强调飞书集成平台（iPaaS）的价值：拖拽式编排，降低对接成本

### 4.3 数据流向
- 简图：数据流入 → AI处理层 → 飞书存储 → 数据流出
